## Project Overview

CrabPOS demonstrates how **a single Rust crate** (`tax_core`) can act as the ground truth for tax calculations and be consumed by two completely different technology stacks **without duplicating any business logic**:

| Layer | Technology | Bridge |
|---|---|---|
| Calculation engine | Rust (`tax_core`) | — |
| Backend API | Ruby on Rails 8 | FFI via `ffi` gem |
| Mobile frontend | Flutter / Dart | FFI via `flutter_rust_bridge` v2 |

All monetary values flow as **`DECIMAL(20, 6)` strings** across every language boundary, so there is zero floating-point drift anywhere in the system.

## Backend — Ruby on Rails

### Prerequisites

- Ruby `3.3+`, Bundler
- Rust toolchain (`rustup`)
- PostgreSQL

### 1. Build the Rust shared library

```bash
cd rust/tax_core
cargo build --release
# Produces:
#   ../../target/release/libtax_core.dylib  (macOS)
#   ../../target/release/libtax_core.so     (Linux)
```

### 2. Install gems and start Rails

```bash
cd rails
bundle install
rails db:create db:migrate
rails server
# API runs on http://localhost:3000
```

### 3. Run the test suite

```bash
cd rails
bundle exec rspec spec/lib/tax_calculator_spec.rb
```

The spec file stubs out the FFI layer, so the compiled `.dylib` / `.so` is **not** required at test time.

### Understanding the Rust Calculation Flow from Rails

```
┌─────────────────────────────────────────────────────────────┐
│                  Ruby on Rails (backend)                     │
│                                                             │
│  TaxCalculator#calculate(price:, region:, category:, …)     │
│        │                                                    │
│        ▼  validate! (ruby-level guard — fast fail)          │
│        │                                                    │
│        ▼  TaxCore.tax_calculate(…)  ← FFI call             │
│              │                                              │
└──────────────┼──────────────────────────────────────────────┘
               │  C ABI — null-terminated DECIMAL(20,6) strings
               │
┌──────────────▼──────────────────────────────────────────────┐
│              Rust  ─  tax_core/src/lib.rs                   │
│                                                             │
│  tax_calculate()  #[no_mangle] extern "C"                   │
│    1. Decimal::parse(price_str)      — fixed-point parse    │
│    2. TaxRegion::from_str(region)    — enum lookup          │
│    3. ProductCategory::from_str(cat) — enum lookup          │
│    4. calculate(&TaxInput { … })                            │
│         a. validate price >= 0                              │
│         b. resolve effective_rate(region, category)         │
│         c. tax_amount  = price × rate   (fixed-point mul)   │
│         d. price_after_tax = price + tax_amount             │
│    5. Serialize all fields back to DECIMAL(20,6) strings    │
│    6. Return CTaxResult { status: 0, … }                    │
└─────────────────────────────────────────────────────────────┘
               │  CTaxResult struct  (by value across FFI)
               │
┌──────────────▼──────────────────────────────────────────────┐
│  Ruby  (back in TaxCalculator#calculate)                    │
│    • CTaxResult#decimal_field(:field) → strip NUL → String  │
│    • BigDecimal(string) — exact, no float involved          │
│    • Return TaxResult Data value object                     │
└─────────────────────────────────────────────────────────────┘
```

**Key files to read:**

| File | What to look at |
|---|---|
| `rust/tax_core/src/lib.rs` | `effective_rate()` rate table, `calculate()` core fn, `tax_calculate()` C ABI |
| `rails/lib/tax_calculator.rb` | `TaxCore` FFI bindings, `CTaxResult` struct layout, `TaxCalculator#calculate` |
| `rails/spec/lib/tax_calculator_spec.rb` | Full RSpec coverage (no compiled library needed) |

## Frontend — Flutter

### Prerequisites

- Flutter SDK `3.x`
- Rust toolchain (`rustup`)
- `flutter_rust_bridge_codegen` CLI (`cargo install flutter_rust_bridge_codegen`)

### 1. Build the Flutter–Rust bridge library

```bash
cd rust/flutter_integration
cargo build --release
```

### 2. Run codegen (only needed after changing Rust API)

```bash
cd flutter
flutter_rust_bridge_codegen generate
# Regenerates lib/src/rust/frb_generated.dart
```

### 3. Get packages and run

```bash
cd flutter
flutter pub get
flutter run
```

### Understanding the Rust Calculation Flow from Flutter

```
┌─────────────────────────────────────────────────────────────┐
│                  Flutter / Dart (frontend)                   │
│                                                             │
│  TaxCalculator.calculate(price: Decimal, region:, cat:, …)  │
│        │                                                    │
│        ▼  _toDecimalString(price)  → "99.990000"            │
│        │                                                    │
│        ▼  _bridge.calculateTax(…)  ← FRB async call        │
│              │                                              │
└──────────────┼──────────────────────────────────────────────┘
               │  flutter_rust_bridge v2  (generated FFI glue)
               │  All strings → Rust, result structs → Dart
               │
┌──────────────▼──────────────────────────────────────────────┐
│     Rust  ─  flutter_integration/src/api/tax.rs             │
│                                                             │
│  pub fn calculate_tax(price, region, category, …)           │
│    → delegates directly to  tax_core::calculate()           │
│       (same fixed-point engine used by Rails)               │
└─────────────────────────────────────────────────────────────┘
               │  FRB-generated Dart types  (TaxResult struct)
               │
┌──────────────▼──────────────────────────────────────────────┐
│  Dart  (back in TaxCalculator.calculate)                    │
│    • Decimal.parse(raw.priceBeforeTax)  — exact Decimal     │
│    • Decimal.parse(raw.taxAmount)                           │
│    • Return TaxResult with all DECIMAL(20,6) fields         │
└─────────────────────────────────────────────────────────────┘
```

**Key files to read:**

| File | What to look at |
|---|---|
| `rust/flutter_integration/src/api/` | Public Rust functions exposed to Dart via FRB |
| `rust/tax_core/src/lib.rs` | Same core engine — `effective_rate()`, `calculate()` |
| `flutter/lib/src/tax_calculator.dart` | `TaxCalculator.calculate`, `_toDecimalString`, `TaxResult` |
| `flutter/lib/src/rust/frb_generated.dart` | Auto-generated bridge — do not edit |

## Why Rust at the Core?

- **No floating-point drift** — custom `Decimal(i128)` type with 6-place fixed scale.
- **Single source of truth** — tax logic lives in exactly one place; both Rails and Flutter use identical computed results.
- **Fast** — the calculation is a handful of integer operations; overhead is measured in nanoseconds.
- **Portable** — the C ABI (via `#[no_mangle] extern "C"`) works with any FFI-capable language.
